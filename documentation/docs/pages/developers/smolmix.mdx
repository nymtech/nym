# smolmix

import { Callout } from 'nextra/components'

`smolmix` is a TCP/UDP tunnel over the Nym mixnet. It uses a userspace network stack [`smoltcp`](https://docs.rs/smoltcp/latest/smoltcp/) to provide real `TcpStream` and `UdpSocket` types that work transparently with the async Rust ecosystem — `tokio-rustls`, `hyper`, `tokio-tungstenite`, and anything else built on `AsyncRead + AsyncWrite`.

```text
┌──────────────────────────────────────────────────────────────────┐
│  Your application (TLS, HTTP, WebSocket, DNS, etc.)              │
│       └─ smolmix::TcpStream / UdpSocket                         │
│            └─ smoltcp (userspace TCP/IP)                         │
│                 └─ Nym mixnet → IPR exit gateway → internet      │
└──────────────────────────────────────────────────────────────────┘
```

Traffic exits the mixnet at an [IPR (Internet Packet Router)](/network/infrastructure/exit-services#ip-packet-router) exit gateway — the exit IP is the gateway's, not yours.

## Installation

Add `smolmix` to your `Cargo.toml`:

```toml
[dependencies]
smolmix = "X.Y.Z"
nym-bin-common = { version = "X.Y.Z", features = ["basic_tracing"] }
tokio = { version = "1", features = ["rt-multi-thread", "macros"] }
blake3 = "=1.7.0"  # required pin — see note below
```

<Callout type="warning">
**Temporary pin required.** You must pin `blake3 = "=1.7.0"` to avoid a build failure caused by a transitive `digest` version conflict. This will be resolved in a future release.
</Callout>

`nym-bin-common` is optional but recommended — it sets up `tracing` logging so you can see mixnet connection progress. The tutorials all use it.

**Minimum Rust version:** 1.70+

### From Git

For unreleased changes, import directly from the repository:

```toml
smolmix = { git = "https://github.com/nymtech/nym", branch = "develop" }
```

## Quick start

```rust
let tunnel = Tunnel::new().await?;

// TCP — drop-in replacement for tokio::net::TcpStream
let tcp = tunnel.tcp_connect("1.1.1.1:443".parse()?).await?;

// UDP — drop-in replacement for tokio::net::UdpSocket
let udp = tunnel.udp_socket().await?;
```

See [Tutorial 1](./smolmix/tutorial) for a full walkthrough.

## When to use smolmix

| | smolmix | Stream module | mixFetch | SOCKS client |
|---|---|---|---|---|
| **Layer** | IP (TCP/UDP) | Message (multiplexed streams) | HTTP | TCP (SOCKS proxy) |
| **Controls both sides?** | No — proxy mode | Yes — E2E | No — proxy mode | No — proxy mode |
| **Runtime** | Rust (tokio) | Rust (tokio) | Browser (WASM) | Any (SOCKS-capable) |
| **API** | `TcpStream`, `UdpSocket` | `AsyncRead + AsyncWrite` | `fetch()` drop-in | SOCKS4/5 protocol |
| **Composability** | Full — TLS, HTTP, WebSocket, DNS, etc. stack on top | Byte streams only | HTTP(S) only | Application-dependent |
| **Best for** | Reaching external services from Rust with standard networking | Peer-to-peer / E2E protocols between Nym clients | Browser HTTP requests | Legacy apps with SOCKS support |

## Security model

<Callout type="warning">
Traffic is Sphinx-encrypted inside the mixnet, but between the exit gateway and the remote host it travels as **normal internet traffic**. Always encrypt the final hop — TLS (rustls), Noise Protocol (snow/libp2p), or any authenticated encryption layer.
</Callout>

### What's protected

| Segment | Encryption | Who sees what |
|---|---|---|
| Your machine → mixnet entry | Sphinx (layered) | Entry gateway sees your IP but not the destination |
| Inside the mixnet (5 hops) | Sphinx (layered) | Each mix node only knows prev/next hop |
| Exit gateway (IPR) | Decrypted to raw IP | IPR sees destination IP + port + unencrypted payload |
| IPR → remote host | Whatever you add (TLS, Noise, etc.) | Remote host sees IPR's IP, not yours |

### Trust boundaries

- **You trust the mixnet** to provide unlinkability between sender and receiver. This is enforced cryptographically by Sphinx packet format and mixing.
- **You trust the IPR exit gateway** in the same way you trust a VPN exit or Tor exit node — it can see your raw traffic unless you encrypt it. The difference is that the IPR doesn't know *who* is sending the traffic (the mixnet hides your identity).
- **Encryption closes the gap.** When you layer TLS (as in [Tutorial 1](/developers/smolmix/tutorial)) or Noise Protocol, the IPR sees encrypted bytes going to a destination IP. It knows you're talking to `httpbin.org:443` but can't read the content.

### Comparison with other privacy tools

| | smolmix | Tor | VPN |
|---|---|---|---|
| **Exit node sees traffic?** | Yes (encrypt it) | Yes (encrypt it) | Yes (encrypt it) |
| **Exit node knows sender?** | No (mixnet hides identity) | No (onion routing) | Yes (VPN provider knows) |
| **Timing analysis resistance** | Strong (mixing, cover traffic) | Weak (low-latency) | None |
| **UDP support** | Yes | No (TCP only) | Yes |

## Performance

The mixnet adds latency at each hop due to mixing delays (packets are held and reordered to prevent timing analysis). Expect:

- **TCP round-trip:** 1–5 seconds for typical requests (vs. milliseconds on clearnet)
- **UDP round-trip:** 1–3 seconds per datagram
- **Throughput:** suitable for API calls, DNS, and moderate file transfers — not bulk streaming

These trade-offs are inherent to the privacy guarantees. The mixing delay is what makes traffic analysis resistant — without it, an observer could correlate packets by timing.

## Examples

Runnable examples in [`smolmix/core/examples/`](https://github.com/nymtech/nym/tree/max/mixtcp/smolmix/core/examples). Each is self-contained and compares clearnet vs. mixnet performance.

```sh
cargo run -p smolmix --example <name>
```

All examples accept `--ipr <ADDRESS>` to target a specific exit node (pass a `Recipient` address to `Tunnel::builder().ipr_address()`).

| Example | Source | What it demonstrates |
|---|---|---|
| TCP | [`tcp.rs`](https://github.com/nymtech/nym/blob/max/mixtcp/smolmix/core/examples/tcp.rs) | HTTPS request via hyper + tokio-rustls — fetches Cloudflare's `/cdn-cgi/trace` to show that the exit IP differs from clearnet |
| UDP | [`udp.rs`](https://github.com/nymtech/nym/blob/max/mixtcp/smolmix/core/examples/udp.rs) | DNS lookup via hickory-proto — sends a raw UDP query to `1.1.1.1:53` through the mixnet |
| WebSocket | [`websocket.rs`](https://github.com/nymtech/nym/blob/max/mixtcp/smolmix/core/examples/websocket.rs) | WebSocket echo via tokio-tungstenite + tokio-rustls — shows the full TCP → TLS → WebSocket stack composing over smolmix |
