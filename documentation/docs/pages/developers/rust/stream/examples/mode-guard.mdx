# Mode Guard

> Source: [`examples/stream_mode_guard.rs`](https://github.com/nymtech/nym/blob/develop/sdk/rust/nym-sdk/examples/stream_mode_guard.rs)

This example demonstrates the mutual exclusion between message-based and stream-based APIs on a `MixnetClient`. Once stream mode is activated, the message API is permanently disabled.

## Why the guard exists

The `MixnetClient` has a single `reconstructed_receiver` channel for inbound messages. In message mode, `wait_for_messages()` reads from it. In stream mode, the router task takes exclusive ownership of it to dispatch incoming data to the correct stream.

If both modes ran concurrently, the router and `wait_for_messages()` would race for the same messages, causing data loss. The guard prevents this.

## Message API works before stream mode

Before calling `listener()` or `open_stream()`, the message-based API works normally:

```rust
let mut client = mixnet::MixnetClient::connect_new().await.unwrap();
let our_address = *client.nym_address();

// This works fine
client
    .send_plain_message(our_address, "hello via message API")
    .await
    .unwrap();
```

## Activating stream mode

Calling `listener()` (or `open_stream()`) activates stream mode. This is a one-way transition:

```rust
let _listener = client.listener().unwrap();
// stream_mode is now true — no going back
```

## Message API fails after activation

```rust
let result = client
    .send_plain_message(our_address, "this should fail")
    .await;

match result {
    Err(Error::StreamModeActive) => {
        // Expected — message API is blocked
    }
    _ => unreachable!(),
}
```

## Split senders share the guard

`client.split_sender()` returns a `MixnetClientSender` that shares the same `stream_mode` flag via an `Arc<AtomicBool>`. So split senders are also blocked:

```rust
let sender = client.split_sender();
let result = sender
    .send_plain_message(our_address, "this also fails")
    .await;
// Also returns Err(Error::StreamModeActive)
```

## Running it

```bash
cargo run --example stream_mode_guard
```
