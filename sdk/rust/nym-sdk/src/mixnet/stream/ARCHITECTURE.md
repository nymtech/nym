# Stream Multiplexing — Architecture

## Overview

The stream subsystem gives each `MixnetClient` the ability to hold many
concurrent byte-oriented channels (`AsyncRead + AsyncWrite`) to different
remote peers, multiplexed over a single client.

```text
┌─────────────────────────────────────────────────────────┐
│                      MixnetClient                       │
│                                                         │
│  ┌──────────────┐   ┌──────────────┐                    │
│  │ MixnetStream │   │ MixnetStream │  ...               │
│  │  (peer A)    │   │  (peer B)    │                    │
│  └──────┬───────┘   └──────┬───────┘                    │
│         │writes            │writes                      │
│         ▼                  ▼                            │
│  ┌─────────────────────────────────┐                    │
│  │     ClientInput.input_sender    │                    │
│  │     (PollSender<InputMessage>)  │                    │
│  └──────────────┬──────────────────┘                    │
│                 │                                       │
│                 ▼                                       │
│           ── mixnet ──                                  │
│                 │                                       │
│                 ▼                                       │
│  ┌─────────────────────────────────┐                    │
│  │     reconstructed_receiver      │                    │
│  │  (taken exclusively by router)  │                    │
│  └──────────────┬──────────────────┘                    │
│                 │                                       │
│                 ▼                                       │
│  ┌─────────────────────────────────┐                    │
│  │           Router task           │                    │
│  │  decode header → dispatch by ID │                    │
│  └──┬──────────────────────────┬───┘                    │
│     │ Open messages            │ Data messages          │
│     ▼                          ▼                        │
│  ┌──────────────┐   ┌──────────────────┐                │
│  │MixnetListener│   │ StreamMap lookup │                │
│  │  .accept()   │   │ → per-stream tx  │                │
│  └──────────────┘   └──────────────────┘                │
└─────────────────────────────────────────────────────────┘
```

## Components

### Wire Protocol (`protocol.rs`)

Every mixnet message body carrying stream data has a fixed 10-byte header:

```text
[Version: 1 byte][StreamId: 8 bytes][MessageType: 1 byte][payload: N bytes]
```

- **Version** — protocol version byte (`STREAM_PROTOCOL_VERSION = 1`).
  Allows future wire-format evolution; unknown versions are rejected on decode.
- **StreamId** — random `u64` (big-endian), generated by the stream opener.
- **MessageType** — `Open` (0) or `Data` (1).

The header is represented by `MixStreamHeader` and decoded messages are
returned as `MixStreamFrame` (header + borrowed payload slice).

No `Close` variant exists. Without message sequencing, a close message can
race ahead of in-flight data. Streams clean up locally via `Drop`. See the
doc comment on `StreamMessageType` for more detail - if we want proper EOF
/ stream closing, we need to implement message ordering.

### Initialization (`ensure_init`)

Stream mode is lazily initialized on the first call to `open_stream()` or
`listener()`. Initialization is a one-way transition:

```text
MixnetClient (message mode)
        │
        │  first open_stream() or listener() call
        │
        ▼
MixnetClient (stream mode)
        │
        ├─ stream_mode = true  (blocks message-based APIs)
        ├─ reconstructed_receiver swapped for dummy
        ├─ Router task spawned with real receiver
        └─ StreamState stored on client
```

Once stream mode is active:
- `wait_for_messages()` / `poll_next()` return `None`
- `send()` / `send_plain_message()` return `Error::StreamModeActive`
- The send-side guard prevents silent message loss (responses would
  arrive at the router, fail stream-header decoding, and be dropped)

**Receiver handoff:** The `reconstructed_receiver` field is
`Option<ReconstructedMessagesReceiver>`. When `ensure_init` activates
stream mode it calls `.take()`, giving the router exclusive ownership
and leaving `None` behind. The `stream_mode` flag (set immediately
before the take) causes all message-mode receive paths to bail early,
so the `None` is never reached in normal operation.

### Router Task (`run_router`)

A background `tokio::spawn`'d loop that:

1. Reads batches of `ReconstructedMessage`s from the receiver
2. Attempts to decode the stream header on each message
3. Dispatches based on message type:

```text
incoming message
       │
       ▼
  decode_stream_message()
       │
       ├─ None → trace log, drop (not a stream message)
       │
       ├─ Open → send InboundOpen to MixnetListener channel
       │         (includes stream_id, sender_tag, initial payload)
       │
       └─ Data → look up stream_id in StreamMap
                  ├─ found  → forward payload to stream's channel
                  │            (remove entry if receiver dropped)
                  └─ missing → silently drop
```

The router shuts down when the `CancellationToken` fires (via
`StreamState::Drop`) or the receiver channel closes.

### Opening a Stream (outbound)

```text
caller
  │
  ▼
open_stream(client, recipient, reply_surbs)
  │
  ├─ ensure_init(client)        ← lazy init if needed
  ├─ generate random StreamId
  ├─ create mpsc channel (tx, rx)
  ├─ insert tx into StreamMap
  ├─ encode Open message with stream header
  ├─ wrap in InputMessage::new_anonymous(recipient, wire, reply_surbs, ...)
  ├─ send via client_input      ← if send fails, remove from StreamMap
  │
  ▼
MixnetStream {
    id, destination: Address { recipient, reply_surbs },
    client_input, streams, inbound_rx: rx, ...
}
```

The peer receives the Open message. If they have a `MixnetListener`, their
router delivers it as an `InboundOpen`.

### Accepting a Stream (inbound)

```text
MixnetListener::accept()
  │
  ▼
  wait on inbound_rx (fed by router's Open dispatch)
  │
  ├─ receive InboundOpen { stream_id, sender_tag, initial_data }
  ├─ reject if sender_tag is None (can't reply without it)
  ├─ create mpsc channel (tx, rx)
  ├─ insert tx into StreamMap
  │
  ▼
MixnetStream {
    id, destination: Anonymous { sender_tag },
    client_input, streams, inbound_rx: rx,
    read_buf: initial_data, ...
}
```

The inbound stream uses `sender_tag` (reply SURBs) to send data back,
since it doesn't know the opener's Nym address directly.

### Data Flow on an Established Stream

**Writing (`AsyncWrite::poll_write`):**
```text
user writes bytes
  │
  ▼
encode_stream_message(id, Data, bytes)    ← prepend 10-byte header
  │
  ▼
make_input_message(wire)                  ← wrap in InputMessage
  │                                          (new_anonymous or new_reply
  │                                           depending on Destination)
  ▼
client_input.input_sender
  .poll_ready() / .start_send()           ← poll-based, not async
  │
  ▼
  into the mixnet
```

**Reading (`AsyncRead::poll_read`):**
```text
  mixnet delivers reconstructed message
  │
  ▼
  Router decodes header, finds stream_id in StreamMap
  │
  ▼
  forwards raw payload via stream's mpsc::UnboundedSender
  │
  ▼
  MixnetStream::poll_read
  │
  ├─ if read_buf has leftover data → drain that first
  ├─ else poll inbound_rx for next chunk
  │    ├─ fits in caller's buffer → copy directly
  │    └─ too large → copy what fits, spill rest into read_buf
  │
  ▼
  bytes delivered to caller
```

### Stream Cleanup

- **`Drop` on `MixnetStream`** — calls `StreamMap::remove_background`
  (spawns a task) to deregister the stream. The router will silently
  drop any further messages for this stream ID.
- **`AsyncWrite::poll_shutdown`** — same removal via `remove_background`,
  but sets `deregistered` so `Drop` doesn't double-remove.
- **`Drop` on `StreamState`** — cancels the router task via
  `CancellationToken`.
- **Idle timeout** — each `StreamEntry` tracks `last_activity`
  (updated on every successful `send_to_stream`). The router runs a
  periodic cleanup (every 60 s) that removes streams idle longer than
  the configured timeout (default: 30 minutes). Configurable via
  `MixnetClientBuilder::with_stream_idle_timeout()`. This prevents
  leaked entries when the remote side drops without the local side
  knowing.

### Shared State (`StreamMap`)

A struct wrapping `Arc<tokio::sync::Mutex<HashMap<StreamId, StreamEntry>>>`.
Each `StreamEntry` holds the channel sender and a `last_activity` timestamp.
Focused methods:

- `register_stream(id)` — creates a channel pair, inserts the sender,
  returns the receiver
- `remove(id)` — async removal
- `remove_background(id)` — sync-safe: spawns a task for the removal
  (used in `Drop` and `poll_shutdown` where `.await` is unavailable)
- `send_to_stream(id, data)` — dispatches data, updates `last_activity`,
  auto-removes if the receiver has been dropped
- `cleanup_stale(max_idle)` — removes entries idle longer than `max_idle`

The struct is `Clone` (inner `Arc`). Callers never touch the lock directly.

- **Router** calls `send_to_stream` to dispatch Data messages
- **Streams** call `remove_background` on drop/shutdown
- **Listener** calls `register_stream` for accepted inbound streams

## Known Limitations

- **No message ordering** — the Mixnet does not guarantee message ordering.
  Large writes spanning multiple Sphinx packets may arrive out of order.
  There is no `Close` message type (a close could race ahead of in-flight
  data). Sequencing is planned for a future release.

- **No protocol discriminator** — there is currently no way to distinguish
  a stream message from a regular Mixnet message. Sending to a non-stream
  client delivers raw bytes with the stream header prepended. A protocol
  discriminator is planned for a future release.
